📊 Os dados de risco

Os benefícios de vibe coding são reais — e os riscos também. Dados de pesquisas independentes e relatos de líderes técnicos revelam um quadro que exige atenção antes de qualquer implementação em produção.

Dados de risco documentados

16 de 18 CTOs consultados sobre vibe coding em produção reportaram desastres — código em produção quebrando sistemas, dados perdidos ou expostos
2,74x mais vulnerabilidades de segurança em código gerado por IA sem revisão estruturada (Veracode Research, 2025)
45% do código gerado por IA contém pelo menos uma vulnerabilidade de segurança conhecida quando avaliado por SAST
-19% de velocidade em tarefas complexas — o risco de aplicar vibe coding onde ele é contraproducente (METR 2025)

💡 O contexto correto

Esses dados não invalidam o vibe coding — eles definem as condições nas quais ele deve ser aplicado. Sistemas de aviação têm altas taxas de acidente quando operados sem treinamento. A ferramenta não é o problema; a ausência de processo é.

💥 Os incidentes reais

Casos documentados públicamente mostram o que acontece quando vibe coding é aplicado sem governança. Esses não são hipotéticos — são relatos de empresas reais com usuários reais afetados.

Lovable — 18.000 usuários expostos

Dados

Um bug no código gerado por IA da plataforma Lovable expôs dados de 18.000 usuários. O problema estava na lógica de autenticação gerada automaticamente, que tinha uma falha de controle de acesso. A empresa levou horas para identificar a causa raiz porque nenhum dev tinha lido o código gerado.

Base44 — Falha de autenticação em produção

Segurança

Plataforma de no-code vibe coding lançou feature com falha de controle de acesso que permitia usuários acessar dados de outras contas. O código tinha sido gerado por IA e publicado sem revisão de segurança. A vulnerabilidade ficou em produção por 72 horas antes de ser detectada.

Replit — Banco de dados deletado em desenvolvimento

Dados

Desenvolvedor usando Replit Agent para automação instruiu o agente a "limpar dados de teste". O agente interpretou o comando de forma mais ampla e deletou o banco de dados de desenvolvimento inteiro. Backup existia, mas 8 horas de trabalho foram perdidas. Caso documentado pelo próprio usuário no X.

⚖️ Propriedade intelectual

A questão de propriedade intelectual em código gerado por IA é uma das mais complexas e menos resolvidas do direito de tecnologia em 2025-2026. Há três dimensões que seu time jurídico precisa conhecer.

Quem é dono do código gerado?

A maioria dos Termos de Serviço das ferramentas de IA (OpenAI, Anthropic, Cursor) concede ao usuário os direitos sobre o output gerado. Porém, nos EUA, código sem autoria humana suficientemente original pode não ser protegível por direito autoral. O USCO (US Copyright Office) já recusou pedidos de registro de obras 100% geradas por IA.

Risco de código de terceiros no output

LLMs foram treinados em código público — incluindo código com licenças restritivas (GPL, AGPL). Há risco de que o modelo gere código que reproduza trechos com licença incompatível com seu uso comercial. Ferramentas como GitHub Copilot têm filtros para isso; outras, não.

Código-fonte como segredo comercial

Ao enviar código-fonte proprietário para um LLM externo para que ele o modifique, você pode estar comprometendo a confidencialidade do seu segredo comercial. Verifique a política de retenção de dados de cada ferramenta — algumas usam os dados para treinar modelos futuros.

4

📋 LGPD e código de IA

A Lei Geral de Proteção de Dados cria responsabilidades específicas quando LLMs são usados no contexto de desenvolvimento de software que processa dados pessoais de usuários brasileiros.

O que NÃO fazer (risco LGPD)

  • ✗Enviar dados pessoais reais para LLMs externos em prompts de desenvolvimento
  • ✗Usar dados de produção (com CPFs, emails, nomes) em contexto de IA sem anonimização
  • ✗Descrever vulnerabilidades específicas de usuários em prompts para a IA corrigir

O que fazer (conformidade)

  • ✓Usar dados sintéticos ou anonimizados em todos os contextos de desenvolvimento com IA
  • ✓Escolher ferramentas com política de não retenção de dados (enterprise tiers)
  • ✓Documentar no ROPA (Registro de Operações de Tratamento) o uso de IA no desenvolvimento
5

🔒 Política de uso seguro

Uma política de uso seguro não precisa ser um documento de 50 páginas. Um checklist de duas categorias — "pode enviar" e "nunca enviar" — já previne a maioria dos incidentes.

PODE enviar para LLMs externos

  • ✓Código novo sem dados de produção
  • ✓Código de projetos open source ou públicos
  • ✓Dados sintéticos criados para testes
  • ✓Descrições abstratas de problema sem identificadores
  • ✓Código de ferramentas internas sem dados sensíveis

NUNCA enviar para LLMs externos

  • ✗Chaves de API, tokens de autenticação, senhas
  • ✗CPFs, emails, dados pessoais de clientes reais
  • ✗Dados financeiros de clientes (saldos, transações)
  • ✗Código-fonte de sistemas classificados como segredo comercial
  • ✗Arquivos de configuração com credenciais de produção
6

🏗️ Arquitetura de governança

A governança eficaz é construída em camadas — cada camada é uma linha de defesa que impede que problemas cheguem ao usuário final. Não é burocracia; é engenharia de processo.

1

Camada 1: Política de uso

Documento de uma página com "pode enviar / não pode enviar". Distribuído e assinado por todos os devs antes de usar ferramentas de IA.

2

Camada 2: Pull Request obrigatório

Todo código (gerado por IA ou não) deve passar por PR antes do merge. Checklist específico para código de IA: "Você revisou todo o código? SAST passou? Dados sensíveis verificados?"

3

Camada 3: SAST no pipeline de CI/CD

Análise estática automatizada em todo código antes do deploy. Vulnerabilidades críticas bloqueiam o pipeline automaticamente. Resultado documentado no PR.

4

Camada 4: Revisão de senior obrigatória

Para código em sistemas de produção críticos, revisão de pelo menos um engenheiro sênior é obrigatória antes do merge. Não pode ser bypassada.

5

Camada 5: Monitoramento e rollback

Alertas de produção configurados para detectar anomalias rapidamente após deploys. Processo de rollback documentado e testado regularmente.

7

🚫 Quando NÃO usar vibe coding

Existem contextos onde o risco de vibe coding é estruturalmente muito alto — não por falta de governança, mas pela natureza do sistema. Nesses contextos, o código deve ser escrito, revisado e validado por humanos com expertise específica no domínio.

💳

Sistemas Financeiros

Lógica de processamento de pagamentos, cálculo de juros, transferências, reconciliação contábil. Um bug pode resultar em perda financeira direta e passivo regulatório.

🏥

Sistemas de Saúde

Dosagem de medicamentos, histórico médico, diagnósticos, sistemas de suporte à decisão clínica. Falhas têm impacto direto na saúde e vida de pacientes.

⚡

Infraestrutura Crítica

Sistemas de energia, água, telecomunicações, controle industrial (SCADA). Falhas podem ter impacto em escala de população e são irreversíveis no curto prazo.

💡 O critério decisivo

Se uma falha no sistema pode resultar em perda irreversível — de vida, de dados críticos, de dinheiro em escala, de infraestrutura — o custo de verificação exaustiva cancela o benefício de velocidade do vibe coding. Nesses casos, a velocidade correta é a velocidade segura.
