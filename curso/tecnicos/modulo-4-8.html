👥 Vibe Coding em Times

Governança, processos e métricas para usar vibe coding com segurança em equipes profissionais. As lições de 16 CTOs que tiveram problemas sérios — e como evitá-los.

1

👤 O dev sênior como reviewer — IA gera, humano revisa

O modelo que funciona em times é claro: agentes implementam, sêniors revisam. Não porque a IA não seja capaz — mas porque o humano traz julgamento sobre contexto de negócio, arquitetura de longo prazo e implicações de segurança que o agente não pode inferir do código.

🔄 O Fluxo que Funciona

1
Dev junior ou agente autônomo implementa feature baseada no ticket
2
CI/CD automático: lint, testes, SAST rodam. PR só avança se tudo passa.
3
Dev sênior revisa focando em: arquitetura, segurança, lógica de negócio
4
Aprovação humana explícita antes do merge em main
5
Deploy automático após aprovação. Monitoramento pós-deploy.

💡 O Reviewer Precisa Entender o Código

Revisores que não entendem o código que estão revisando são uma vulnerabilidade de governança. "Aprova tudo que o CI passou" não é revisão — é rubber stamp. Sêniors precisam conseguir identificar problemas de lógica de autorização, falhas de segurança sutis e violações de padrões arquiteturais.

2

🛠️ Múltiplas ferramentas por papel — IDE certa para cada função

Times de vibe coding profissional não usam uma ferramenta para tudo. A escolha de ferramenta por papel e tipo de tarefa otimiza produtividade e reduz riscos. Padronizar a ferramenta por tipo de trabalho, não por preferência individual, é a política que funciona.

Papel / Tarefa Ferramenta Recomendada Por quê
Desenvolvimento diárioCursorContexto de projeto, multi-arquivo, extensões
Automação e CI/CDClaude CodeTerminal-first, ideal para pipelines e scripts
Protótipos rápidosWindsurf ou LovableVelocidade máxima, revisão depois
Compliance / EnterpriseGitHub Copilot EnterpriseIsolamento de dados, auditoria, conformidade SOC 2
Análise arquiteturalClaude.ai (Opus)Raciocínio profundo sem contexto de IDE

📋 Política de Ferramentas no Handbook

Documente as ferramentas aprovadas pelo time no handbook de engenharia. Inclua: quais são aprovadas, para qual tipo de tarefa, e quais são proibidas (ex: ferramentas que enviam código para servidores não auditados). Consistência de ferramentas reduz surface de ataque e facilita onboarding.

3

🔒 PRs obrigatórios — nenhum código de IA vai para main sem review

Esta é a política de governança mais importante para vibe coding em times: nenhum código — gerado por IA ou por humano — vai para a branch principal sem passar por review e CI/CD. Sem exceções. Nem para "hotfixes urgentes".

✓ Política de Branch Protection

  • ✓Require pull request reviews (mínimo 1 aprovação)
  • ✓Require status checks (CI deve passar)
  • ✓Require branches to be up to date
  • ✓Restrict who can push to main (apenas tech lead)
  • ✓Include administrators (sem exceções)

Configuração no GitHub

# Settings → Branches → Branch rules
# Branch name pattern: main

[✓] Require a pull request
    Min approvals: 1
[✓] Require status checks:
    - CI / quality
    - CI / security
[✓] Require branches up to date
[✓] Include administrators
[✓] Allow force pushes: NUNCA

⚠️ O "Hotfix Urgente" que Destruiu o Banco

Um dos casos mais comuns de desastre com vibe coding: CTO faz push direto em main para "resolver urgente em produção". O agente gerou código que parecia correto mas tinha um bug crítico que o CI teria capturado. Sem review, sem CI — sem proteção. A política sem exceções existe exatamente para momentos de pressão.

4

📋 Padronização de prompts e regras — CLAUDE.md compartilhado

Em um time, cada desenvolvedor com seu próprio conjunto de regras e prompts resulta em codebase inconsistente — o mesmo problema do patchwork, mas em escala de time. Regras e prompts devem ser assets de equipe, versionados no repositório.

🗂️ Estrutura de Arquivos de Time

repositorio/
├── CLAUDE.md              # Regras globais para Claude Code
├── .cursorrules           # Regras globais para Cursor
├── docs/
│   ├── prompts/
│   │   ├── feature.md     # Template para novas features
│   │   ├── bugfix.md      # Template para bugfixes
│   │   ├── refactor.md    # Template para refatoração
│   │   └── security-review.md  # Checklist de segurança
│   ├── decisions/         # ADRs (Architecture Decision Records)
│   └── ai-guidelines.md   # Política de uso de IA no time

💡 Templates de Prompts como Padrão de Time

Crie templates de prompt para os tipos de tarefa mais comuns (feature, bugfix, refactor) e coloque em docs/prompts/. Quando um dev precisa implementar uma feature, começa com o template e adapta. Consistência de input gera consistência de output do agente.

5

🔐 Testes de segurança no processo — SAST/DAST integrado

Segurança como processo contínuo, não como fase final. SAST analisa código estático; DAST (Dynamic Application Security Testing) testa a aplicação em execução. Ambos devem ser automáticos e obrigatórios no processo de desenvolvimento com IA.

SAST (Estático — antes do deploy)

  • • CodeQL — em cada PR automaticamente
  • • Semgrep — regras customizadas do projeto
  • • npm audit — vulnerabilidades em dependências
  • • truffleHog — secrets em commits

Custo: gratuito (ferramentas open-source no CI)

DAST (Dinâmico — em staging)

  • • OWASP ZAP — scan automático de aplicação rodando
  • • Burp Suite Community — análise manual de endpoints
  • • Nuclei — templates de vulnerabilidades conhecidas
  • • Penetration testing — anual para sistemas críticos

Custo: ferramentas gratuitas + tempo para configuração

📋 Política: Security Gate no Pipeline

Configure o pipeline para bloquear merges quando SAST detecta vulnerabilidades de alta severidade. Médias geram alertas mas não bloqueiam. Baixas são registradas para revisão futura. O time decide os thresholds — mas a automação aplica consistentemente.

📊 Medindo produtividade — métricas que realmente importam

Métricas incorretas criam incentivos perversos. "Linhas de código por dia" foi inútil antes da IA; com IA é ainda mais inútil. As métricas que importam medem resultado de negócio e qualidade de software, não velocidade de geração de código.

📊 Framework DORA Adaptado para Vibe Coding

Métrica O que mede Target Elite
Deployment FrequencyCom que frequência vai a produçãoMúltiplos/dia
Lead Time for ChangesTicket → deploy em produção<1 hora
Change Failure Rate% deploys que causam incidente<5%
Time to RestoreIncidente → serviço restaurado<1 hora
Test CoverageCobertura de testes automatizados>80%

Métricas úteis com IA

  • • Lead time para features (ticket → produção)
  • • Taxa de bugs por feature em produção
  • • Cobertura de testes e tendência
  • • Taxa de aceitação de código de IA em review
  • • Custo de API por feature entregue

Métricas inúteis ou prejudiciais

  • • Linhas de código por dia (IA infla artificialmente)
  • • Número de commits (agentes fazem commits granulares)
  • • Velocidade de sprint (incentiva code dumping)
  • • Features completadas sem considerar bugs introduzidos
7

💀 O que deu errado — 16 de 18 CTOs reportaram desastres

Em pesquisa com 18 CTOs que adotaram vibe coding sem estrutura de governança, 16 reportaram problemas sérios. Os padrões de falha são consistentes e preveníveis — mas apenas se você os conhecer antes de cometê-los.

Padrão 1: Velocidade sem processo (8/16 CTOs)

"Velocidade de vibe coding sem PRs ou review. Em 3 meses, o codebase estava inutilizável — ninguém entendia o que o sistema fazia, nenhum teste, bugs críticos de segurança em produção."

Solução: implementar PRs e CI/CD desde o primeiro dia, não "quando o projeto crescer".

Padrão 2: Júniores sem supervisão (5/16 CTOs)

"Demos vibe coding para desenvolvedores juniores e os deixamos trabalhar autonomamente. Eles aceitavam tudo que a IA gerava sem entender. Resultado: dívida técnica massiva e vulnerabilidades de segurança que levamos meses para limpar."

Solução: sênior como reviewer obrigatório. Juniors aprendem com IA, sêniors validam resultado.

Padrão 3: Sem contexto de arquitetura (3/16 CTOs)

"Cada dev usava IA com seus próprios prompts. O codebase ficou inconsistente: 4 padrões diferentes de autenticação, 3 frameworks de validação, nomenclatura caótica. Refactoring levou mais tempo do que construir do zero."

Solução: CLAUDE.md e .cursorrules compartilhados no repositório desde o início.

✅ O que os 2 CTOs sem desastres fizeram diferente

  • ✓Processo antes de velocidade: implementaram PRs, CI/CD e code review antes de usar IA extensivamente
  • ✓Arquitetura documentada: CLAUDE.md com regras claras antes de dar IA para o time
  • ✓Sêniors revisando: nunca delegaram revisão para júniores sem supervisão
  • ✓Gradualidade: piloto com 1 time, aprendeu, ajustou processo, depois expandiu
  • ✓Treinamento real: não assumiram que "é intuitivo". Treinaram o time em prompting e revisão de código de IA.
