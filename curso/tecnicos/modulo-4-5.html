🛡️ Por que testes são o guardrail do vibe coding

Existe uma relação direta entre cobertura de testes e a capacidade de usar IA com segurança: quanto mais testes, mais seguramente o agente pode operar autonomamente. Um agente com suite robusta de testes pode refatorar, otimizar e adicionar features com confiança — porque qualquer regressão é detectada imediatamente.

📊 O Impacto Real dos Testes em Projetos de IA

67%
de redução em regressões quando agente tem suite de testes robusta (Devin, 2025)
4x
mais rápido para aprovar PRs de agentes com >80% de cobertura de testes
90%

💡 Testes como Especificação

Em vibe coding avançado, testes não são apenas verificação — são especificação. Um teste bem escrito descreve exatamente o comportamento esperado do sistema. O agente pode usar essa especificação para implementar corretamente e para verificar se sua implementação está correta.

2

🧩 Testes unitários por prompt — cobertura com edge cases

A IA é excepcionalmente boa em gerar testes unitários — especialmente para identificar e cobrir edge cases que humanos frequentemente esquecem. Com o prompt certo, você obtém uma suite completa incluindo casos extremos em minutos.

📋 Prompt para Testes Unitários Completos

"Gere testes unitários completos para a função abaixo.

@src/utils/calculateDiscount.ts

Requisitos dos testes:
1. Casos felizes: desconto válido aplicado corretamente
2. Edge cases:
   - Desconto 0% (sem desconto)
   - Desconto 100% (produto gratuito)
   - Desconto > 100% (deve lançar erro)
   - Preço negativo (deve lançar erro)
   - Preço zero (deve lançar erro)
   - Tipos incorretos (null, undefined, string)
3. Precisão: teste que R$ 0.001 é arredondado corretamente

Use Vitest, não Jest.
Cada teste deve ter descrição clara em português.
Organize em describe blocks por categoria de teste."

💡 Peça Edge Cases Explicitamente

Sem instrução explícita, a IA gera testes para o "happy path" apenas. Sempre peça explicitamente: "inclua edge cases", "teste entradas inválidas", "teste valores extremos (zero, negativo, máximo possível)". A IA conhece os edge cases — mas precisa de permissão para incluí-los.

3

🔗 Testes de integração — testando fluxos completos

Testes de integração verificam que componentes funcionam corretamente quando conectados. Em APIs REST, isso significa testar endpoints reais com banco de dados real (ou banco de testes). A IA gera esses testes de forma muito eficaz quando você fornece o contrato esperado da API.

🔗 Exemplo: Teste de Integração para API de Pedidos

describe('POST /orders', () => {
  beforeEach(async () => {
    await db.orders.deleteMany();
    await createTestUser();
  });

  it('cria pedido com produtos válidos', async () => {
    const response = await request(app)
      .post('/orders')
      .set('Authorization', `Bearer ${testToken}`)
      .send({ items: [{ productId: 'p1', qty: 2 }] });

    expect(response.status).toBe(201);
    expect(response.body).toMatchObject({
      id: expect.any(String),
      status: 'pending',
      total: expect.any(Number)
    });

    // Verifica que foi salvo no banco
    const order = await db.orders.findFirst();
    expect(order?.userId).toBe(testUserId);
  });

  it('retorna 401 sem autenticação', async () => {
    const response = await request(app)
      .post('/orders')
      .send({ items: [] });
    expect(response.status).toBe(401);
  });

  it('retorna 422 com payload inválido', async () => {
    const response = await request(app)
      .post('/orders')
      .set('Authorization', `Bearer ${testToken}`)
      .send({ items: [] }); // carrinho vazio
    expect(response.status).toBe(422);
  });
});

🎭 Testes E2E — Playwright/Cypress gerado por IA

Testes End-to-End verificam fluxos completos do ponto de vista do usuário — clicando em botões, preenchendo formulários, navegando entre páginas. A IA pode gerar testes Playwright ou Cypress a partir de descrições de comportamento em linguagem natural.

🎭 Prompt para Teste E2E

"Gere um teste Playwright para o fluxo
de checkout do e-commerce.

Fluxo:
1. Usuário está na página de produto
2. Clica em 'Adicionar ao carrinho'
3. Vê feedback visual que item foi adicionado
4. Navega para /cart
5. Verifica que produto aparece com preço correto
6. Clica em 'Finalizar compra'
7. Preenche endereço de entrega
8. Seleciona forma de pagamento (cartão)
9. Confirma pedido
10. Vê página de confirmação com número do pedido

Assunções:
- Use seletores por data-testid, nunca por CSS
- Inclua setup: criar usuário de teste e fazer login
- Inclua teardown: limpar pedidos criados
- Timeout de 30s para operações de rede"

💡 data-testid é Essencial

Instrua a IA a sempre usar seletores data-testid em vez de classes CSS ou texto. Classes mudam com redesigns; texto muda com traduções; data-testid é estável. Adicione no CLAUDE.md: "Todo componente de UI deve ter data-testid para seletores de teste."

5

🔄 TDD invertido — testes primeiro, implementação depois

O TDD invertido é a técnica de vibe coding mais poderosa para features críticas: você escreve (ou pede para a IA escrever) os testes que definem o comportamento esperado antes da implementação, então pede para o agente implementar o código que faz os testes passarem.

Passo 1: Escrever os testes

"Escreva os testes para uma
função validatePassword que:
- Aceita senhas com 8+ chars
- Exige maiúscula e minúscula
- Exige número
- Rejeita espaços
- Retorna erros específicos

NÃO implemente a função ainda,
apenas os testes."

Passo 2: Implementar para passar nos testes

"Agora implemente a função
validatePassword em TypeScript
de forma que os testes em
@src/utils/password.test.ts
passem completamente.

Execute npm run test para
verificar antes de finalizar."

📊 Vantagens do TDD Invertido

  • • Especificação precisa: testes definem o contrato antes da implementação
  • • Verificação automática: o agente sabe exatamente quando terminou (testes passando)
  • • Menos retrabalho: ambiguidades são descobertas na fase de testes, não de implementação
  • • Cobertura garantida: 100% de cobertura dos comportamentos documentados nos testes
6

📈 Cobertura de código — métricas e como atingi-las com IA

Cobertura de código mede que percentual das linhas, branches e funções do código são exercitados pelos testes. 80% é o threshold mínimo profissional. Abaixo disso, o agente opera em território desconhecido onde pode introduzir regressões sem detecção.

⚙️ Configuração de Cobertura com Vitest

// vitest.config.ts
export default defineConfig({
  test: {
    coverage: {
      provider: 'v8',
      reporter: ['text', 'html', 'lcov'],
      thresholds: {
        lines: 80,      // falha CI se abaixo de 80%
        functions: 80,
        branches: 70,
        statements: 80
      },
      exclude: [
        'src/migrations/**',
        'src/types/**',
        '**/*.test.ts'
      ]
    }
  }
})

📋 Prompt para Aumentar Cobertura

"Execute npm run test:coverage e analise
o relatório. O arquivo @src/services/OrderService.ts
tem apenas 45% de cobertura.

Identifique os branches não cobertos
e gere testes adicionais para atingir 85%+.

Foco especial em:
- Branches de erro (catch blocks)
- Condicionais de negócio (if/else)
- Fluxos alternativos de cancelamento e reembolso"

💡 Cobertura de Qualidade vs. Cobertura de Quantidade

100% de cobertura não significa zero bugs. Um teste que apenas verifica "não lança exceção" não testa comportamento real. Instrua a IA: "testes devem verificar o comportamento esperado, não apenas que o código executa sem erro." Asserções explícitas > execução sem falhas.

🔍 Debugging sistemático — isolando problemas para o agente

Debugging eficaz com IA requer isolamento do problema antes de enviar para o agente. O contexto mínimo viável para debugging é a habilidade mais valiosa: quanto menor e mais preciso o problema isolado, maior a probabilidade de resolução na primeira tentativa.

✗ Debugging ineficaz

Enviar o sistema completo com "alguma coisa está errada no checkout"

  • • Agente analisa código desnecessário
  • • Múltiplas trocas de mensagem
  • • Sugestões vagas e genéricas
  • • Alta chance de solução incorreta

✓ Debugging eficaz

Teste que falha + código mínimo que reproduz o problema

  • • Agente vê exatamente o problema
  • • Resolução em 1-2 mensagens
  • • Solução específica e precisa
  • • Teste confirmando a correção

🔬 Técnica: Teste de Regressão como Ponto de Partida

// 1. Escreva um teste que falha
it('deve calcular desconto progressivo corretamente', () => {
  // Este teste está falhando — use-o como guia
  expect(calculateDiscount(100, 0.1, 0.05)).toBe(85);
  // atual retorna 85.5 — erro de arredondamento?
});

// 2. Envie para o agente com o contexto:
// "@src/utils/discount.ts
// O teste acima está falhando.
// Esperado: 85, recebendo: 85.5.
// Analise a lógica de arredondamento e corrija."
