🚢 CI/CD e Deploy Profissional

Com vibe coding, CI/CD passa de conveniência para necessidade. Cada linha de código gerada por IA precisa passar por um pipeline de qualidade antes de chegar à produção.

⚙️ Por que CI/CD é ainda mais importante com vibe coding

Em desenvolvimento tradicional, CI/CD captura bugs que humanos introduzem por descuido. Em vibe coding, CI/CD captura bugs que agentes introduzem por falta de contexto ou instrução ambígua. O pipeline é a verificação objetiva que o agente não pode fazer por si mesmo.

🔄 Pipeline Mínimo Profissional

1
Lint — ESLint, TypeScript strict mode, formatação Prettier
2
Testes unitários — cobertura mínima 80%, falha se abaixo
3
SAST — Semgrep ou CodeQL para vulnerabilidades de segurança
4
Build — verificação que o projeto compila sem erros
5
Deploy automático — apenas se todos os steps anteriores passarem
2

⚙️ GitHub Actions básico — pipeline gerado por IA

O GitHub Actions é a ferramenta padrão de CI/CD para a maioria dos projetos de vibe coding por integrar nativamente com o repositório, ser gratuito para projetos privados (2.000 min/mês) e ser facilmente gerado por IA. Peça para o agente criar o workflow completo.

⚙️ Workflow Completo para Node.js + TypeScript

# .github/workflows/ci.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'

      - run: npm ci

      - name: Lint
        run: npm run lint

      - name: Type check
        run: npx tsc --noEmit

      - name: Tests com cobertura
        run: npm run test:coverage
        env:
          DATABASE_URL: ${{ secrets.TEST_DATABASE_URL }}

      - name: Upload cobertura
        uses: codecov/codecov-action@v4

  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Semgrep SAST
        uses: semgrep/semgrep-action@v1
        with:
          config: p/security-audit p/owasp-top-ten

  deploy:
    needs: [quality, security]
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Deploy to Vercel
        run: npx vercel --prod --token=${{ secrets.VERCEL_TOKEN }}

🔍 SAST no pipeline — análise de segurança em cada PR

SAST no pipeline transforma segurança de uma fase final para um processo contínuo. Em vez de encontrar vulnerabilidades depois do deploy, você as encontra antes de qualquer código chegar à produção.

CodeQL (GitHub nativo)

- uses: github/codeql-action/init@v3
  with:
    languages: javascript-typescript
    queries: security-extended
- uses: github/codeql-action/analyze@v3

Gratuito, integrado ao GitHub, detecta vulnerabilidades complexas de múltiplos arquivos.

Semgrep (regras customizáveis)

- uses: semgrep/semgrep-action@v1
  with:
    config: >-
      p/security-audit
      p/secrets
      p/owasp-top-ten
      p/javascript

Regras customizáveis, mais rápido que CodeQL, excelente para padrões específicos do projeto.

💡 SAST não Substitui Code Review

SAST detecta padrões conhecidos de vulnerabilidade. Não detecta falhas de lógica de autorização, regras de negócio incorretas ou RLS ausente. Use SAST como primeira linha, code review humano como segunda. Ambos são necessários.

🚀 Deploy automático com Vercel/Railway

Vercel e Railway são as plataformas de deploy padrão para projetos de vibe coding por sua integração nativa com GitHub, zero configuração de infraestrutura e deploy automático a cada push. Cada merge em main dispara deploy automático.

Critério Vercel Railway
Melhor paraFrontend, Next.js, funções serverlessAPIs, workers, banco de dados, workers
Banco de dadosNão (use Supabase/Neon externo)PostgreSQL, Redis nativos

PreçoGratuito até 100GB/mês$5/mês (Hobby)

🔑 Preview Deploys: A Killer Feature

O Vercel cria automaticamente um deploy de preview para cada PR. Isso significa que revisores podem testar o código de IA em ambiente real antes de aprovar o merge — sem precisar rodar localmente. Para vibe coding em times, esta é uma funcionalidade crítica.

5

🚩 Feature flags — liberando código de IA gradualmente

Feature flags permitem deploar código de IA em produção mas ativá-lo apenas para subconjuntos controlados de usuários. Se algo der errado, você desliga o flag sem rollback de código. É a estratégia de release mais segura para vibe coding.

🚩 Implementação Simples de Feature Flag

// feature-flags.ts
export const flags = {
  newCheckoutFlow: process.env.FF_NEW_CHECKOUT === 'true',
  aiRecommendations: process.env.FF_AI_RECS === 'true',
};

// Uso no componente
if (flags.newCheckoutFlow) {
  return ;  // código de IA
}
return ;  // código estável

// Liberação gradual com Vercel
// Environment Variables: FF_NEW_CHECKOUT=true
// Rollback: alterar para false, sem novo deploy

Estratégia de Release Gradual

  • • 1%: equipe interna (dogfooding)
  • • 5%: usuários beta voluntários
  • • 25%: release parcial com monitoramento
  • • 100%: release completo após 48h estável

Ferramentas de Feature Flag

  • • Variáveis de ambiente: simples e grátis para flags binários
  • • PostHog: gratuito, flags + analytics integrados
  • • LaunchDarkly: enterprise, regras complexas por usuário
6

⏪ Rollback automático — revertendo quando métricas degradam

Rollback automático é a capacidade de reverter um deploy automaticamente quando métricas de saúde do sistema degradam além de um threshold. Em projetos de vibe coding, onde código gerado pode ter bugs sutis que só aparecem em produção, rollback automático é segurança crítica.

⏪ Rollback no Vercel

# Via CLI — rollback imediato para deploy anterior
vercel rollback [deployment-url]

# Via GitHub Actions — rollback automático por métricas
- name: Check Error Rate
  run: |
    ERROR_RATE=$(curl -s "https://api.sentry.io/...")
    if [ "$ERROR_RATE" -gt "5" ]; then
      echo "Error rate acima de 5% — iniciando rollback"
      vercel rollback --token=${{ secrets.VERCEL_TOKEN }}
    fi

💡 Métricas para Acionar Rollback

  • • Taxa de erro HTTP 5xx acima de 2% (baseline: <0.1%)
  • • Latência P95 acima de 2s (baseline: <500ms)
  • • Taxa de conversão caiu mais de 20% vs. 7 dias anteriores
  • • Erros de JavaScript no browser acima de 5%
7

📡 Monitoramento pós-deploy — Sentry, logs e alertas

Código gerado por IA pode ter bugs que passam por todos os checks mas só aparecem com tráfego real e dados de produção. Monitoramento pós-deploy é o último guardrail do processo de vibe coding profissional.

Sentry — Error Tracking

  • • Captura exceções em tempo real
  • • Stack traces com contexto de usuário
  • • Alertas por Slack/email no primeiro erro
  • • Gratuito até 5K erros/mês

Logs Estruturados

  • • Pino.js para Node.js (alta performance)
  • • Logs em JSON estruturado
  • • LogTail, Datadog ou CloudWatch
  • • Correlação com request ID

Métricas de Negócio

  • • PostHog para analytics de produto
  • • Funnels de conversão
  • • Feature flag analytics
  • • Alertas de anomalia

⚙️ Configuração Básica do Sentry

// sentry.config.ts
import * as Sentry from "@sentry/nextjs";

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.NODE_ENV,
  tracesSampleRate: 0.1, // 10% das transações
  beforeSend(event) {
    // Remove dados sensíveis antes de enviar
    if (event.request?.data) {
      delete event.request.data.password;
      delete event.request.data.creditCard;
    }
    return event;
  },
});
