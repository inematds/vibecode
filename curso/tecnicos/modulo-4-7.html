💻 Claude Code como agente autônomo

Claude Code é o agente mais acessível para desenvolvedores individuais e times pequenos. Com suporte a sessões longas, CLAUDE.md como memória de projeto e capacidade de executar comandos, criar arquivos e fazer commits, é um agente completo de desenvolvimento acessível via terminal.

💻 Sessão Autônoma de Claude Code

# Iniciar sessão autônoma
claude --model opus-4 --dangerously-skip-permissions

# Exemplo de task longa delegada:
> Analise o codebase em @src/. Implemente o módulo
> de relatórios conforme especificado em @docs/reports-spec.md.
>
> Requisitos:
> - Siga os padrões definidos no CLAUDE.md
> - Execute npm run test após cada módulo implementado
> - Commit separado para cada módulo funcional
> - Pare e me consulte se encontrar decisão arquitetural
>   não coberta pela spec
>
> Comece pelo schema de banco e me mostre o plano
> antes de implementar.

# Claude Code então:
# 1. Lê o CLAUDE.md (regras do projeto)
# 2. Lê a spec de relatórios
# 3. Apresenta plano para aprovação
# 4. Implementa módulo por módulo
# 5. Executa testes após cada módulo
# 6. Faz commits incrementais
# 7. Para e reporta quando finaliza

💡 CLAUDE.md é a Memória do Agente

Em sessões longas, o Claude Code lê o CLAUDE.md no início de cada sessão. Isso garante que o agente sempre respeite as regras do projeto, mesmo que a sessão anterior tenha terminado. É a diferença entre um agente que "esquece" o contexto e um que mantém coerência ao longo do tempo.

🔗 Arquitetura multi-agent — orquestra de especialistas

Em vez de um agente tentando fazer tudo, a arquitetura multi-agent distribui responsabilidades entre agentes especializados que trabalham em paralelo. O resultado é maior qualidade, maior velocidade e menor custo por output — porque cada agente usa o modelo mais adequado para sua tarefa específica.

🏗️ Arquitetura Multi-Agent para Feature Completa

P
Agente de Planejamento Claude Opus

Recebe o requisito, decompõe em subtarefas, define dependências, cria plano de execução e distribui para agentes de implementação.

I1
Agente Backend Claude Sonnet

Implementa endpoints, lógica de negócio, queries de banco. Paralelo com Agente Frontend.

I2
Agente Frontend Claude Sonnet

Implementa componentes, integração com API, estados de loading/error. Em paralelo com Backend.

T
Agente de Testes Claude Haiku

Gera e executa testes unitários, integração e E2E para outputs dos agentes de implementação.

R
Agente de Revisão Claude Opus

Revisa código por segurança, consistência e qualidade. Gera relatório para aprovação humana final.

🔌 MCP na prática — conectando agentes ao mundo real

MCP é o que dá aos agentes acesso ao ambiente real de produção. Sem MCP, o agente trabalha com código em texto. Com MCP, ele pode interagir com banco de dados, APIs, infraestrutura e ferramentas externas como se fosse um desenvolvedor humano.

Servidores MCP Essenciais

  • @supabase/mcp-server-supabase — queries, schema, auth, storage
  • @modelcontextprotocol/server-github — PRs, issues, repositório
  • @modelcontextprotocol/server-filesystem — leitura/escrita de arquivos
  • @modelcontextprotocol/server-postgres — banco PostgreSQL direto
  • mcp-server-sentry — erros de produção, alertas

Casos de Uso com MCP

  • "Verifique os erros do Sentry das últimas 24h e corrija os top 3 por frequência"
  • "Crie um PR no GitHub com as mudanças e solicite review"
  • "Query o banco de produção para entender o schema atual"
  • "Leia todos os arquivos em src/api/ e gere a documentação OpenAPI"

⚠️ Princípio de Menor Privilégio no MCP

Configure permissões mínimas necessárias. Um agente de desenvolvimento não precisa de acesso de escrita ao banco de produção. Use ambientes separados (staging vs. production). Configure MCP para produção apenas com read-only access para diagnóstico.

🔬 OpenHands/OpenDevin — alternativa open-source

OpenHands (anteriormente OpenDevin) é o projeto open-source mais avançado de agente de software autônomo. Permite rodar localmente ou em sua própria infraestrutura — sem enviar código para serviços externos. Ideal para experimentação e para compliance que não permite código em nuvem.

🔬 Setup do OpenHands Local

# Instalação via Docker
docker pull docker.all-hands.dev/all-hands-ai/runtime:0.38-nikolaik

docker run -it --rm \
  -e SANDBOX_RUNTIME_CONTAINER_IMAGE=docker.all-hands.dev/all-hands-ai/runtime:0.38-nikolaik \
  -e LOG_ALL_EVENTS=true \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v ~/.openhands-state:/.openhands-state \
  -p 3000:3000 \
  --add-host host.docker.internal:host-gateway \
  --name openhands-app \
  docker.all-hands.dev/all-hands-ai/openhands:0.38

# Acesse http://localhost:3000
# Configure seu API key (Claude, OpenAI, etc.)
# O agente tem acesso ao filesystem e executa comandos

Vantagens do OpenHands

  • • 100% open-source (MIT License)
  • • Código nunca sai da sua infraestrutura
  • • Suporta qualquer modelo via API
  • • Ideal para pesquisa e experimentação

Limitações vs. Devin/Claude Code

  • • Requer infraestrutura própria para produção
  • • Menos otimizado para projetos grandes
  • • Suporte community (não enterprise)
  • • Estabilidade menor que soluções comerciais

👑 Você como orquestrador — o novo papel do engenheiro sênior

Karpathy declarou em fevereiro de 2026 que o próximo passo após vibe coding é "Agentic Engineering". O papel do engenheiro sênior está evoluindo de implementador para arquiteto de sistemas de agentes. As habilidades mudam, mas a necessidade de expertise técnica sólida permanece — e se torna ainda mais crítica.

🔄 Evolução do Papel do Engenheiro Sênior

2020-2024 (tradicional)

  • • Escreve código de alta qualidade diretamente
  • • Revisa PRs linha por linha
  • • Mentora juniores em implementação
  • • Toma decisões arquiteturais em reuniões
  • • Debug de problemas complexos manualmente

2025-2026 (agentic engineering)

  • • Define arquitetura que agentes respeitam
  • • Revisa PRs de agentes por segurança e lógica
  • • Mentora em como orquestrar agentes efetivamente
  • • Toma decisões que definem o que agentes podem fazer
  • • Debug de comportamento de agentes, não de código

💡 As Habilidades que Permanecem Valiosas

  • Arquitetura de sistemas: agentes seguem a arquitetura que você define. Engenheiros sem esse conhecimento produzem patchwork em escala.
  • Segurança: agentes reproduzem vulnerabilidades sem instrução explícita. Conhecimento de segurança é mais crítico, não menos.
  • Julgamento técnico: distinguir quando o código do agente é correto vs. plausível mas errado. Requer entender código profundamente.
  • Decomposição de problemas: a habilidade de decompor sistemas complexos em subtarefas que agentes podem executar bem.

💡 A Paradoxo da Expertise em 2026

Os engenheiros que mais se beneficiam de agentes são os mais experientes — porque sabem exatamente o que pedir, como validar o resultado e quando não confiar. Engenheiros menos experientes aceitam código incorreto que "parece certo". A IA amplifica expertise, não a substitui.
