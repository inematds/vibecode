📊 Os dados: 45% do código de IA tem falhas de segurança

Antes de qualquer técnica, é essencial entender a escala do problema com dados concretos. O código gerado por IA não é inerentemente inseguro — mas ele reproduz os padrões de segurança (e insegurança) que existem nos dados de treinamento.

📊 Dados de Pesquisa (2024-2025)

  • 45% do código gerado por IA contém pelo menos uma falha de segurança — Veracode State of Software Security 2025
  • 2,74x mais vulnerabilidades em código gerado por IA vs. código escrito por humanos — CodeRabbit Analysis 2025
  • 40% dos desenvolvedores admitem não revisar código de IA antes de fazer commit — Stack Overflow Survey 2025
  • 78% das vulnerabilidades em código de IA são categorizadas como OWASP Top 10 — CVE Database 2025
  • 16/18 CTOs reportaram problemas de segurança sérios com vibe coding não supervisionado — Pesquisa CTO Council 2025

⚠️ Por que a IA comete erros de segurança

A IA não tem intenção maliciosa — ela aprende padrões dos dados de treinamento. Exemplo: muito código de tutorial online não valida input corretamente. A IA aprende e reproduz esse padrão. O modelo não sabe a diferença entre código de produção e código educacional.

Além disso, a IA otimiza para "funcionar" — não para "funcionar com segurança". Segurança requer instrução explícita em cada contexto.

🔟 OWASP Top 10 para código de IA

O OWASP Top 10 lista as vulnerabilidades mais críticas de aplicações web. A IA cria todas elas, mas as mais frequentes têm padrões identificáveis que você pode detectar e prevenir com instruções específicas nos arquivos de regras.

A01: SQL Injection

Código vulnerável (IA gera assim sem instrução):

// PERIGOSO - SQL Injection
const query = `SELECT * FROM users
WHERE email = '${email}'`;

Instrução no CLAUDE.md para prevenir:

// SEGURO - Parameterizado
const user = await prisma.user.findUnique({
  where: { email } // Prisma sanitiza
});

A03: XSS — Cross-Site Scripting

Vulnerável:

// PERIGOSO
element.innerHTML = userInput;

Seguro:

// SEGURO
element.textContent = userInput;
// ou sanitize com DOMPurify

A01 (API): Broken Object Level Authorization (BOLA)

Vulnerável (MUITO comum em código de IA):

// PERIGOSO - qualquer user acessa
app.get('/orders/:id', async (req) => {
  return db.orders.findById(req.params.id);
});

Seguro:

// SEGURO - verifica ownership
app.get('/orders/:id', auth, async (req) => {
  return db.orders.findFirst({
    where: { id: req.params.id,
             userId: req.user.id } // CRÍTICO
  });
});

📋 Instrução no CLAUDE.md para Segurança

## Regras de Segurança (OBRIGATÓRIAS)
- SEMPRE verificar req.user.id ao acessar dados do usuário
- NUNCA confiar em parâmetros de URL sem validar ownership
- SEMPRE usar ORM parametrizado (Prisma) - nunca SQL raw com input do usuário
- NUNCA retornar stack traces em produção
- SEMPRE validar e sanitizar input com Zod antes de processar
- NUNCA usar innerHTML com dados de usuário

🛡️ Row Level Security — o erro mais comum em apps Lovable

Row Level Security (RLS) é uma funcionalidade do PostgreSQL/Supabase que garante que usuários só acessam seus próprios dados no nível do banco de dados, independente do que a aplicação faça. É a última linha de defesa e a mais frequentemente esquecida em vibe coding.

🚨 O Caso dos 18.000 Usuários (Lovable, 2025)

Um app construído com Lovable (plataforma de vibe coding) foi exposto por um pesquisador de segurança: sem RLS configurado, qualquer usuário autenticado conseguia acessar dados de qualquer outro usuário fazendo requests diretos à API do Supabase — que o Lovable expõe automaticamente.

18.000 usuários tiveram dados potencialmente expostos. O problema: Lovable não ativava RLS por padrão, e a IA não adicionava políticas de RLS sem instrução explícita.

🛡️ RLS Obrigatório em Cada Tabela

-- Habilitar RLS em TODA tabela nova
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

-- Policy: usuário só vê seus próprios dados
CREATE POLICY "users_own_orders"
ON orders
FOR ALL
USING (auth.uid() = user_id);

-- Policy: admins veem tudo
CREATE POLICY "admins_all_orders"
ON orders
FOR ALL
USING (
  EXISTS (
    SELECT 1 FROM profiles
    WHERE id = auth.uid() AND role = 'admin'
  )
);

💡 RLS no CLAUDE.md

Adicione no seu CLAUDE.md: "SEMPRE criar políticas RLS ao criar novas tabelas no Supabase. Nunca criar tabela sem RLS habilitado. Padrão mínimo: usuário só acessa linhas onde user_id = auth.uid()." Uma instrução que previne o incidente dos 18K usuários.

🔑 Gestão de secrets — never hardcode, never commit

Secrets hardcodados é um dos erros mais comuns e mais devastadores em código de IA. O modelo aprende de exemplos online onde credenciais frequentemente aparecem hardcodadas para simplificação. Você precisa instruir explicitamente contra isso.

✗ O que a IA pode gerar sem instrução

// NUNCA fazer isso
const stripe = new Stripe(
  'sk_live_abc123xyz...'
);

const db = postgres({
  password: 'minhasenha123'
});

✓ Correto

// SEMPRE assim
const stripe = new Stripe(
  process.env.STRIPE_SECRET_KEY!
);

const db = postgres({
  password: process.env.DB_PASSWORD
});

🔐 Checklist de Gestão de Secrets

  • ✓.env no .gitignore — sempre, sem exceção
  • ✓.env.example com chaves vazias — no repositório para documentar variáveis necessárias
  • ✓git-secrets ou trufflehog — scan automático de commits para detectar secrets acidentais
  • ✓GitHub Secret Scanning — Alertas automáticos se um secret detectado é pushado
  • ✓Rotate imediatamente — se um secret foi commitado, rotacione antes de corrigir o commit

🔍 SAST: análise estática automatizada

SAST (Static Application Security Testing) analisa o código sem executá-lo, detectando vulnerabilidades antes do deploy. Em um fluxo de vibe coding, SAST deve rodar automaticamente em cada commit ou PR — não manualmente depois.

Ferramentas SAST Gratuitas

  • CodeQL — GitHub, gratuito para open source, detecta vulnerabilidades complexas
  • Semgrep — regras customizáveis, roda no CI ou localmente
  • Bandit — específico para Python, rápido e preciso
  • ESLint Security Plugin — detecta padrões inseguros em JavaScript/TypeScript

Ferramentas SAST Pagas

  • Snyk — integra com IDE, CI/CD e SCM. Foco em dependências
  • Veracode — análise profunda, compliance. Enterprise.
  • SonarQube — qualidade de código + segurança. Muito adotado em times
  • Checkmarx — foco em conformidade e compliance regulatório

⚙️ Semgrep no GitHub Actions

- name: Run Semgrep
  uses: semgrep/semgrep-action@v1
  with:
    config: >-
      p/security-audit
      p/secrets
      p/owasp-top-ten
  env:
    SEMGREP_APP_TOKEN: ${{ secrets.SEMGREP_APP_TOKEN }}

👁️ Code review focado em segurança

Mesmo com SAST automatizado, code review humano focado em segurança é insubstituível. O humano detecta falhas de lógica de negócio que ferramentas automatizadas não conseguem — como autorização incorreta que tecnicamente compila mas semanticamente está errada.

📋 Checklist de Code Review para Código de IA

Autenticação e Autorização

  • Todas as rotas privadas têm middleware de auth?
  • Verificação de ownership em todos os recursos?
  • Tokens expiram corretamente?

Input e Output

  • Todo input de usuário é validado com schema?
  • Queries usam ORM ou parâmetros?
  • Erros não expõem stack trace ou dados internos?

Secrets e Dados

  • Nenhum secret hardcodado?
  • Dados sensíveis não estão sendo logados?
  • RLS configurado em tabelas Supabase?

Dependências

  • Novas dependências auditadas com npm audit?
  • Versões pinadas no package.json?
  • Dependências conhecidas e confiáveis?

💀 Casos reais de incidentes — o que aconteceu e por quê

Estudar casos reais é a forma mais eficaz de internalizar a importância de segurança. Estes não são cenários hipotéticos — são incidentes documentados que afetaram usuários reais de plataformas de vibe coding.

Lovable — 18.000 usuários expostos

2025

Problema: RLS não configurado por padrão. API do Supabase exposta publicamente permitia qualquer usuário autenticado acessar dados de qualquer outro.

Lição: RLS não é opcional em apps Supabase. Habilite em cada tabela. Verifique que a API key do Supabase no frontend é a anon key, nunca a service key.

CurXecute — CVE-2025-54135

2025

Problema: Plugin de IDE para vibe coding executava código arbitrário via prompt injection. Um prompt malicioso poderia executar comandos no sistema do desenvolvedor.

Lição: Cuidado com plugins de IDE de origem desconhecida. Agentes com acesso ao filesystem e executor de código são vetores de ataque. Revise permissões de cada plugin.

Replit — banco de dados deletado em produção

2025

Problema: Agente de IA interpretou "limpe os dados de teste" como instrução para executar DELETE sem WHERE em banco de produção. Sem backup recente.

Lição: Agentes não devem ter acesso irrestrito a ambientes de produção. Use ambientes separados, backups automáticos e confirmação humana antes de operações destrutivas.

Base44 — vazamento de dados de clientes

2025

Problema: Código gerado por IA para exportação de relatórios não validava que o usuário tinha acesso aos dados solicitados. Qualquer usuário podia exportar dados de qualquer empresa.

Lição: BOLA (Broken Object Level Authorization) é a vulnerabilidade #1 em código de IA. Sempre verifique ownership. "Funciona" não é o mesmo que "funciona com segurança".
